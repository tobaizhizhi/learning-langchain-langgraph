# 课程大纲生成规则

版本日期：2026-06-22

用途：以后生成或修改每个阶段课程大纲时，必须先遵守这份规则，避免提前工程化、概念堆叠和“看起来专业但当前没必要”的内容。

## 最高原则

每个学习内容都必须回答 4 个问题：

1. 这是 LangChain / LangGraph 官方核心概念吗？
2. 这是当前代码马上要用的吗？
3. 这是未来作品集明确会复用的吗？
4. 如果删掉它，当前学习是否受影响？

根据答案给内容打标签：

```text
必须学：官方核心，或当前马上使用，删掉会影响学习。
值得学：不是官方核心，但当前项目或作品集明确复用。
知道即可：未来可能用，但当前不影响主线。
暂缓：当前不用，删掉不影响学习。
删除：没有明确价值，只增加理解负担。
```

## 每个阶段必须写清楚

每个阶段大纲必须包含：

- 阶段目标：一句话说明这一阶段到底练什么。
- 必须学：不掌握就无法进入下一步的内容。
- 值得学：对项目质量有帮助，但不是官方核心。
- 暂缓内容：现在先不学，避免分心。
- 最小可运行成果：这一阶段完成后能跑什么命令或 demo。
- 验收标准：怎么判断真的学会了。
- 作品集关系：这部分未来怎么进入作品集。

## 每个 Step 必须写清楚

每个 Step 必须用下面格式：

```md
### Step N：标题

做什么：

为什么现在做：

价值标签：必须学 / 值得学 / 知道即可 / 暂缓

重点看：

练习：

验收：

不做什么：
```

其中 `不做什么` 必须明确写出当前不该扩展的范围。

## 代码实现规则

### 1. 先用最小直接实现

能用一段清晰代码说明概念，就不要先抽象。

优先：

```ts
await model.invoke([
  { role: "system", content: systemPrompt },
  { role: "user", content: userPrompt },
]);
```

暂缓：

```ts
PromptBuilder
PromptRegistry
PromptPipeline
```

### 2. 抽象必须通过删除测试

新增模块前先问：

```text
如果删掉这个模块，当前学习会不会受影响？
如果不会，就不要把它放进主线。
```

允许提前放一个很小的结构，但必须标明：

```text
这是未来铺垫，不是当前必学。
```

### 3. 两个真实用例之前不抽通用模块

如果只有一个用例，直接写。

如果同类逻辑出现两次，可以考虑函数。

如果出现三次以上，才考虑更正式的模块。

### 4. 不把未来项目复杂度提前搬进当前阶段

例如：

- benchmark 属于模型能力实验台，不要在 messages 入门阶段深入实现。
- prompt template 属于复用工具，不要在刚学 system/user 时强行实现。
- LangSmith evaluation 属于评估阶段，不要在第一次 invoke 时展开。
- LangGraph state 属于状态图阶段，不要在模型接口阶段提前建复杂 state。

## 文档写法规则

### 1. 每个概念必须标注来源

使用下面分类：

```text
官方核心概念
工程实践
作品集准备
可选扩展
暂缓
```

不要把工程实践讲得像官方必修。

### 2. 先讲最小心智模型，再讲扩展

顺序必须是：

```text
最小能跑
为什么这样
什么时候不够
再怎么扩展
```

不要反过来。

### 3. 避免“概念连环套”

如果一个新概念需要再解释 3 个新概念才能听懂，就先不要放进当前 Step。

### 4. 每个 Step 最多引入一个新抽象

例如：

- Step 学 `ModelConfig`，就不要同时引入 fallback、价格表、model registry。
- Step 学 `messages`，就不要同时引入 prompt builder、dataset、evaluator。

### 5. 必须写“现在不用深学”

对未来有用但当前不必掌握的内容，要明确写：

```text
现在只需要知道它存在，不需要实现。
```

## 以后生成课程大纲前的检查清单

生成前先检查：

- 是否有官方核心概念被遗漏？
- 是否把可选工程实践写成了必修？
- 是否有当前不会马上用的代码模块？
- 是否有“为了以后可能用”而提前写的大抽象？
- 是否每个 Step 都有明确的“不做什么”？
- 是否能用一个命令或一个小 demo 验收？
- 是否能解释它和作品集的关系？

如果不能通过这份清单，先收缩大纲，再继续写。
